## Schema Rework â€” Per-Feature Audit

> Every schema should make sense from the user's perspective, not just the MIB's.
> Before gNMI adapter, every feature schema gets audited for sensible API design.

### Process per feature:
1. What does v1 expose? (getter return shape, setter args)
2. What does the user actually want? (global state + table rows, not flat MIB dumps)
3. What does the engine support? (sub_tables, method-scoped attrs, CRUD, key_map, index_filter)
4. Restructure schema to match user intent, using engine primitives

### Example: mrp_sub_ring

**Current** (MIB-shaped):

## VRRP & Compound Index Decomposition

### Engine primitive needed: `index_fields` on GET path

The table getter (`_execute_schema_table`) needs to decompose compound OID walk suffixes into per-field values on the GET path. Currently `index_fields` only works on CREATE/SET.

**Problem**: SNMP walks of compound-indexed tables return suffixes like `"45.1"` (ifIndex.vrid) or `"45.1.112.105.110.103.45.49"` (ifIndex.vrid.impliedString). The engine needs to split these into schema field values.

**What works today**:
- `index_fields` on CREATE/SET encodes compound indexes for writes
- `_encode_index()` handles `oid_suffix`, `field_dict`, `hex_string`, `raw` per protocol YAML
- Method-scoped gather (only gather attrs in current method's defaults + index_fields)
- Skip empty wire/source attrs (index-only fields populated by decomposition)

**What's needed**:
- Schema method declares `index_fields: [interface, vrid]` â€” engine splits walk suffix by `.`
- First N-1 fields = one OID part each (integer), last field = remaining parts
- If wire declares `index_type: implied_string`, last field consumes remaining as ASCII bytes (`112.105.110.103.45.49` â†’ `"ping-1"`)
- Schema attrs with no `wire:`/`source:` (e.g. `interface: {}`, `vrid: {}`) are populated from decomposed suffix, not walked separately
- `key_map: ifindex` remaps first decomposed field to port name, builds `":"` separated compound key (e.g. `"vlan/150:1:ping-1"`)
- Compound keys detected from walked data must match method's attrs (not other tables in same schema)
- Primary_key attr included in row data when compound (not skipped like simple tables)

**Key design rule**: Wire `index_type` drives the decomposition format. Schema declares field names. Engine executes. No magic detection.

**Test cases on GRS1042 (.254)**:
- 2-part: `get_vrrp_instances` â€” `[interface, vrid]` from VRRP-MIB vrrpOperEntry
- 3-part with implied_string: `get_vrrp_tracking` â€” `[interface, vrid, track_name]` from HM2 hm2AgentVrrpTrackingEntry
- All compound data verified correct via raw SNMP walks and MOPS captures
